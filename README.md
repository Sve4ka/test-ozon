### Запуск

Запуск может быть выполнен через команды docker-compose, для удобства были написаны make команды.

```make memory```  - запуск с использованием in-memory хранилища

```docker-compose --env-file .env.memory up --build``` - аналогичная команда

```make postgres``` - запуск с использованием postgres хранилища

```docker-compose --env-file .env.postgres up --build``` - аналогичная команда

Настройка запуска происходит в .env файле, для удобства были написаны готовые .env файлы для различных сценариев.

Для того, чтобы postgres хранилище не поднималось, когда оно не используется было принято решение использовать profiles
в docker-compose, что отделило запуск различных образов в разных сценариях использования.

### Генерация ссылок

Алгоритм генерации 10-значного кода находиться в `/internal/generate/code.go`

Внутри проекта работа идет только с самим кодом короткой ссылки.
Так как требование
`2. Метод Get, который будет принимать сокращённый URL и возвращать оригинальный URL`
можно проичитать двояко, с одной стороны это запрос GET который не должен иметь тело по принятым соглашениям, с другой
стороны требование по получению именно сокращенного URL,
я приняла решение сделать две ручки один метод POST (`/link/short`) второй метод GET (`/link/{code}`).
Для поддержки унификации всех ссылок в .env можно задать в каком формате ожидается получить ссылку, тк хранение
происходит только короткого кода ссылки, то смена адреса не повлияет на работу сервиса.

### Поведение при большом количестве одновременных пользователей

При работе с in-memory памятью возникнет проблема с превышением памяти. В случае же с Postgres такая проблема
маловерятна, однако при большом количестве пользователей будет иметь соответственно большую нагрузку, дабы разгрузить
его я добавила решение с hit-cache на базе Redis. Для запуска с ним можно использовать команду `make cache`.
Это позволяет разгрузить основное хранилище и иметь в быстром доступе часто используемые ссылки.
При не использовании кеша работает заглушка (`/cache/noop`). При любом сценарии кеш не является источником истины, если
он возвращает ошибку на любом этапе его работы влияние на работу приложение оказано не будет, ошибка будет залогирована,
а данные будут поступать из полной базы данных.
